# 決定: ワークスペース3層構成の採用

## 背景
- 1人法人の立ち上げに向けて、個人知識管理・開発・経営統括を一貫して運用する基盤が必要になった
- 既存の Obsidian 保管庫と開発フォルダだけでは、エージェントオーケストレーション・事業横断管理の置き場がなかった

## 選択肢
1. **Obsidian 保管庫に統括機能を統合** → メリット: 一元管理 / デメリット: iCloud同期の制約、保管庫が肥大化
2. **開発フォルダ内にHQ機能を追加** → メリット: 既存構成を流用 / デメリット: 関心の分離ができない
3. **~/hq/ として独立した統括ワークスペースを新設** → メリット: 役割明確、エージェント定義を集中管理 / デメリット: 参照先が3箇所に増える

## 決定
- **採用**: 選択肢3（~/hq/ 新設）
- **理由**: 社長の判断・指示に特化した場所を分離することで、知識蓄積（vault）・実装（dev）・経営（hq）の責務が明確になる。エージェントの権限制御もフォルダ単位で管理しやすい
- **却下した選択肢の理由**: 選択肢1はiCloud制約、選択肢2は責務混在

## 影響範囲
- 影響する事業: 全事業
- 影響するフォルダ: ~/vault/（参照元として）、~/dev/（作業先として）、~/hq/（新設）
- 必要なアクション:
  - ~/hq/ フォルダの作成と全ファイルの配置
  - 各エージェント定義の動作確認
  - Linear プロジェクトの作成（TRADE, BOT, HP, SNS, NOTE, TOOLS, CORP）

## レビュー予定
- 効果測定日: 2026-04-30（1ヶ月運用後）
- 撤回基準: エージェント運用が機能せず手動作業が減らない場合は構成を再検討

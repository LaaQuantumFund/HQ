## 0. このドキュメントの目的

自分がやっていること・やろうとしていることを、仕組みとして人に説明できる言葉にする。
AIエージェントを動かす際の「なぜこう設計したのか」「なぜこう判断するのか」を
構造的に言語化し、思考の基盤とする。

## 1. 市場認識と失敗要因

### 1-1. 市場の現実

Gartner は、2026年末までに企業アプリの40%にAIエージェントが統合されると予測している
（2025年時点では5%、1年で8倍の成長）。
Sam Altman は2024年に「1人で10億ドル企業が可能になる」と予測し、
2026年4月に Matthew Gallagher の Medvi（年間売上$1.8B）がそれを実証した。
参入障壁は「ノートPC1台とエージェンティック・ワークフローの深い理解」にまで下がっている。

### 1-2. なぜ多くの企業がエージェント導入に失敗するのか

Deloitte の2026年レポートによると、失敗の本質は
「AIに何をさせるか」ではなく「人間とAIの役割分担の再設計ができていない」ことにある。
Gartner は2027年末までにAIエージェント・プロジェクトの40%以上がキャンセルされると予測し、
Deloitte はパイロットから本番への成功率をわずか11%と報告している。

### 1-3. 典型的な失敗パターン

- エージェント・ウォッシュ: 既存ツールにAIラベルを貼るだけで、ワークフロー自体は変わっていない
- 壊れたプロセスの自動化: 非効率な業務をそのままAIに移植し、非効率が高速化するだけ
- 推論ドリフト: エージェントが長いタスクチェーンで文脈を失い、意図と異なる方向に進む
- コスト爆発: Opus級モデルを全タスクに使い、API費用が制御不能になる
- サイレント・フェイルア: エージェントがエラーを報告せず、誤った結果を正常として返す

## 2. 第一原理思考 (First Principles Thinking)

### 2-1. 定義

第一原理思考とは、既存の前提や類推を捨て、
「これ以上分解できない基本的事実」まで掘り下げてから再構築する思考法。
Aristotle が「ものごとを知るための最初の基盤」と定義したものを、
現代の経営者が意思決定の中核に据えている。

### 2-2. 事例

Elon Musk (SpaceX): ロケットの見積もりが$65Mだったとき、
「ロケットの原材料費は市場価格の2%に過ぎない」という事実に立ち返り、
自社製造を選択。打ち上げコストを10分の1に削減した。

Jensen Huang (NVIDIA): 「今の条件、今のツール、今の動機を踏まえて、
これをゼロからやり直すならどうするか？」と常に問い続けることで、
GPUをゲーム用からAIインフラの中核へと転換させた。

Johannes Gutenberg: ワイン圧搾機の技術と活版印字を組み合わせ、
印刷機を発明。異なる分野の「第一原理」を結合させたイノベーション。

### 2-3. Laa への適用

AIエージェント組織を設計する際、「既存の企業構造を真似る」のではなく、
「1人の人間が最大のアウトプットを出すために何が必要か」から逆算する。

具体的な問い:
- この作業の本質的な目的は何か？（形ではなく機能を見る）
- もしゼロから始めるなら、今あるツールでどう設計するか？
- 「みんながやっているから」という理由だけで採用しているものはないか？
- この構造の各部品は、本当に必要か？不要な部品はないか？

## 3. 戦略 = 行動 (Strategy is What You Do)

### 3-1. 原則

Jensen Huang: 「戦略とは言葉ではない。戦略とは行動だ。
会社に戦略があっても、人々の行動がそれと違うなら、その戦略は実行されていない。
戦略とは、結局のところ私が言うことではなく、彼らがやることだ。」

### 3-2. 意味

戦略はスライドやドキュメントに書くものではなく、
実際に時間・リソース・脳のエネルギーをどう使っているかそのもの。
「Phase 1 の戦略はトレードで利益を出すこと」と言いながら、
1日の大半をツール設定やインフラ構築に費やしていたら、
実質的な戦略は「ツール設定」になっている。

### 3-3. Laa の日次検証ルール

毎日の振り返りで以下を確認する:
- 今日、最も多くの時間を使った活動は何か？
- それは現在のフェーズの戦略目標に直結しているか？
- 直結していないなら、なぜそうなったのか？
- 明日、戦略に直結する行動に時間を戻すには何を変えるか？

## 4. Delegate-Review-Own フレームワーク

### 4-1. 構造

Delegate（委任）: 定型作業・データ収集・コード生成・文書作成をAIに渡す。
- CTO Agent がコード生成
- CFO Agent が freee 仕訳入力
- CMO Agent が SNS 下書き作成
- n8n が定時ブリーフィング送信

Review（検証）: 人間が品質と方向性を確認する。
- 外部影響あり（顧客・法務・資金）の場合は必ず人間が最終承認
- 外部影響なし（内部文書・テスト・ログ整理）の場合はAIが自律完結

Own（所有）: 社長が最終責任を持つ領域。
- トレードの意思決定（エントリー・エグジットの最終判断）
- 経営判断（事業の方向性、撤退判断、投資配分）
- AIエージェントとワークフローの設計そのもの
- ビジョンの定義と更新

### 4-2. Human-in-the-Loop と Human-Above-the-Loop の違い

Human-in-the-Loop は、エージェントの全ステップを人間が確認するモデル。
安全だが、スケールしない。Phase 1 ではこのモデルで運用し、信頼を構築する。

Human-Above-the-Loop は、人間がビジョンを設定し、
5%の例外処理のみを担当するモデル。
Phase 3 で目指す「Slackで承認のみ」がこれに該当する。

移行は段階的に行う:
Phase 1: すべて手動確認（信頼構築期）
Phase 2: 低リスク領域から自律実行を許可（テスト期）
Phase 3: 高リスク領域のみ人間介入（運用期）

## 5. Solo-CEO パラダイム

### 5-1. 新しい経営者像

「Solopreneur（1人起業家）」の時代は終わり、「Solo-CEO」の時代が始まっている。
違いは、Solopreneur が「1人で全部やる人」なのに対し、
Solo-CEO は「AIエージェント組織を指揮する1人の経営者」であること。

Medvi の Gallagher は、ChatGPT/Claude/Grok でコード生成、
Midjourney/Runway で広告クリエイティブ、ElevenLabs でカスタマーサービスを構築し、
自分は「組み合わせの判断」だけをしていた。
初期投資$20K、構築期間2ヶ月、初年度売上$401M。

### 5-2. Solo-CEO に必要な能力

1. アーキテクチャ設計力: エージェント組織の構造を設計し、役割分担を定義する力
2. Intent Specification: 曖昧な目標を、AIが実行可能な明確な仕様に変換する力
3. 品質判断力: エージェントの出力が「十分に良いか」を素早く判断する力
4. コスト感覚: どのタスクにどのモデル（Haiku/Sonnet/Opus）を使うか判断する力
5. リスク認識: どこに人間の介入が必要かを見極める力

### 5-3. 自分がやるべき5%の定義

Phase 1 において、自分だけが担当すべき領域:
- トレードのエントリー・エグジット判断
- Brain ナレッジの最終承認（draft → confirmed の昇格）
- エージェント・ワークフローの設計変更
- 退職・法人設立などの人生判断
- 学習テーマの優先順位決定

これ以外は、段階的にAIまたは自動化に委任する。

## 6. 直接性と Intent Specification

### 6-1. 原則: Be Direct

Jensen Huang: 「何かの方向性が気に入らなければ、そう言う。」
これは対人だけでなく、AIへの指示にも適用される。

曖昧な指示 → 曖昧な結果。
「ブログ記事を書いて」ではなく、
「10,000 PV を達成するまでドラフト・分析・修正を繰り返せ」と
Goal Persistence（目標持続型指示）を与えることが、
エージェンティックな指示の本質。

### 6-2. Intent Specification の技術

Andrej Karpathy が2026年に提唱した「Agentic Engineering」の核心は、
「コードを書く人」から「意図を仕様化する人」への転換。
Karpathy 自身のワークフローは4週間で
「80%手動コーディング」から「80%エージェント駆動」に変化した。

Intent Specification の構成要素:
- 目的: 何を達成したいのか（機能レベル、数値目標で記述）
- 制約: やってはいけないこと、使ってはいけないリソース
- 品質基準: 「完了」の定義、合格ラインの明示
- コンテキスト: エージェントが知るべき背景情報（CONTEXT.md がこれに該当）
- フィードバック・ループ: 結果が基準を満たさない場合の再試行ルール

### 6-3. 曖昧な指示 vs 仕様化された指示の例

悪い例: 「トレードジャーナルアプリを改善して」
良い例: 「Trade Journal アプリに以下の機能を追加せよ。
(1) エントリー時の感情タグ（5種類から選択）、
(2) ルール遵守チェックリスト（6項目、全チェックでないとエントリー不可）、
(3) 週次サマリーの自動生成（勝率・平均R倍数・ルール違反回数を表示）。
DPGEフローに従い、Evaluator が全項目のテストに合格するまでループせよ。」

## 7. 継続学習ルール

### 7-1. 原則

Jensen Huang: 「私は良い学生であることに多くを捧げている。」
Solo-CEO は「知識の孤立」が最大のリスク。
学ばなくなった時点で、AIエージェントへの指示の質が劣化し始める。

### 7-2. 学習の民主化 = Brain への転記

GenSpark での壁打ち、Twitter で見つけた洞察、動画から得た知見は、
24時間以内に Brain に書かなければ「存在しない知識」と同じ。
Brain に書かれた知識だけが、エージェント組織全体に共有される。

転記ルール:
- GenSpark 壁打ち → 終了後すぐに draft ファイルとして Brain に保存
- 外部情報（Twitter, YouTube, 記事）→ 要点を自分の言葉で書き直して保存
- 自分の気づき・思考 → 日報または該当エリアのファイルに追記
- draft → confirmed への昇格は週次レビューで実施
- Brain に書くのは「検証済みの知識」と「確定した判断」のみ
- 失敗した試行錯誤のプロセスそのものは書かない （ただし「この方法は失敗した」という教訓は書く）
- 理由: 未検証の情報が Brain に蓄積すると、 エージェントがそれを事実として参照し、判断品質が劣化する 
- draft → confirmed の昇格プロセスがこの規律を担保する

### 7-3. 学ぶべき領域の優先順位（Phase 1）

1. トレード手法と市場構造（ICT, リテール心理）→ これが収益の根幹
2. エージェンティック・エンジニアリング → これが事業の基盤
3. 法人設立・税務の基礎知識 → Phase 2 への準備
4. コンテンツ・マーケティング → Phase 3 への種まき

## 8. ワークスペース3層構造

### 8-1. 設計思想

第一原理: 「1人の人間がAIエージェント組織を指揮するために、
情報をどう構造化すべきか？」

答え: 知識（Brain）、指揮（HQ）、実装（Dev）を分離する。
知識と指示が混在すると、エージェントが文脈を誤解する。
指示とコードが混在すると、変更管理が破綻する。

### 8-2. 各層の役割

Brain（~/Brain/）: Obsidian vault。知識の単一ソース。
PARA構造でマインドセット、習慣、トレードルール、ジャーナル、ビジネスプラン等を格納。
iCloud + GitHub 同期。エージェントは COS のみ読取権限。

HQ（~/hq/）: 統括層。エージェント定義（CLAUDE.md）、ガイドライン、
ロードマップ、インフラ手順、自動化フロー、会計、ダッシュボード。
CONTEXT.md もここに配置。

Dev（~/dev/）: 開発層。プロジェクト別 Git リポジトリ。
CLAUDE.md、.mcp.json、エージェント定義、STRUCTURE.md、各プロジェクトフォルダ。

## 9. エージェント組織設計

### 9-1. 設計原則

- 責任範囲の明確化: 役職ごとに担当領域を定義し、プロンプト精度を向上させる
- 権限最小化: 各エージェントは自分の担当領域の情報のみアクセスする（Need-to-Know）
- コスト最適化: Haiku（定型タスク、cost 1x）→ Sonnet（標準、cost 3x）→ Opus（高度判断、cost 15x）
- スケーラビリティ: 新事業追加は HQ にエージェント定義を追加するだけで対応

### 9-2. 組織図

社長（意思決定・長期ビジョン）
  └── CEO Agent（指示解釈・タスク分配）
        ├── COS（リサーチ / Brain 管理）
        ├── CMO（SNS / コンテンツ / SEO）
        ├── CQO（品質管理 / テスト）
        ├── CTO（開発 / EA / インフラ）
        ├── CFO（freee 連携 / 会計 / 予算）
        ├── CIO（市場データ分析 / BI）
        └── CLO（法務 / コンプライアンス）

## 10. 開発手法: DPGE ハーネスエンジニアリング

Design（設計）: Designer エージェントが Figma MCP で UI 設計。
Plan（計画）: Planner が実装仕様書を生成。
Generate（生成）: Generator がコードを実装。
Evaluate（評価）: Evaluator がテスト・照合。未達は該当ステップへリターン。

このループは Karpathy の PEV Loop（Plan-Execute-Verify）と本質的に同じ。
「作って終わり」ではなく「検証に合格するまでループする」ことが鍵。

## 11. 通信フロー

### 11-1. Slack 中心の理由

- 非同期: 社長が常時監視する必要がない
- 監査証跡: すべての指示と結果が記録される
- モバイル完結: iPhone から全業務を確認・承認できる

### 11-2. フロー

社長（Slack で指示）
  → CEO Agent（指示を解釈・分解）
    → CIO Agent（データ取得）
    → COS Agent（情報整理）
    → CTO Agent（実装）
  → CEO Agent（結果を統合・報告）
→ 社長（Slack で承認 or 追加指示）

## 12. 構造的拡張パス

### 12-1. 成長に応じた組織モデルの進化

Phase 1 - Diamond Model（現在）:
1人の Meta-Agent（CEO Agent）+ 数個の専門エージェント。
全タスクを社長が手動で確認。信頼構築期。

Phase 2 - Pod Model:
KPI ごとの自律ユニット（Growth Pod, Product Pod, Finance Pod）。
各 Pod が独立したフィードバック・ループを持つ。
低リスク領域から Human-Above-the-Loop を開始。

Phase 3 - Hierarchical Orchestration:
エージェントがエージェントを管理する多層構造。
CEO Agent がタスクを L2 オーケストレーターに委任し、
L2 が L3 の実行エージェントを指揮する。
社長は Slack で5-10分のブリーフィング・承認のみ。

### 12-2. 拡張時の判断基準

新しい Pod やエージェントを追加する条件:
- 繰り返し発生するタスクが週3回以上ある
- そのタスクの品質基準を明文化できる
- エラー時の影響が限定的で、自律実行を許容できる
- Haiku または Sonnet で処理可能（Opus が必要なら人間が判断すべき）

## 13. 成熟度モデル

Level 1 - Observer: AIツールを試用。手動操作が中心。
Level 2 - Experimenter（Phase 1）: 壁打ち・ナレッジ構築・単発タスク委任。
Level 3 - Integrator（Phase 2）: エージェント組織稼働・一部自律実行。
Level 4 - Pioneer（Phase 3）: AI主導の事業運営・人間は例外処理のみ。
Level 5 - AI Delegation（最終形態）: 複数事業を AI が自律経営、利益を分散投資。

現在は Level 1 → Level 2 の移行期。

## 14. Failure Modes の体系化

### 14-1. 自分が陥りやすい失敗パターン（過去の経験から）

- 分析なしのギャンブルトレード → メンタルと習慣の問題
- ツール設定に没頭して本来の目標を忘れる → 「戦略 = 行動」の違反
- 情報収集が目的化する（Twitter のノイズ）→ 受動的消費者への退行
- 完璧を求めて着手が遅れる → 第一原理で「最小限の機能」を定義すべき

### 14-2. エージェント運用の失敗パターン

- プロンプトベースの順序制御: 「必ずAの後にBをやれ」→ プログラム的に強制すべき
- バッチAPIをブロッキング・ワークフローに使用: 非同期APIの誤用
- サブエージェントのサイレント・フェイルア: エラーを報告しない → ログ監査必須
- ツールのスコープ過大: 1つのツールに多機能を詰め込む → 責任範囲を分割
- 全タスクに Opus を使用: コスト爆発 → Haiku/Sonnet/Opus の使い分け厳守
- コンテキスト・ウィンドウの浪費: 不要な情報をプロンプトに含める → 必要最小限
- レビューなしの自律実行: 信頼構築前に権限を与えすぎる → Phase に応じた段階的委任

## 15. 評価フレームワーク (Review の具体化)

### 15-1. 出力品質の測定基準

- 正確性: 事実誤認がないか
- 完全性: 指示した要素がすべて含まれているか
- 一貫性: 他のドキュメントやルールと矛盾していないか
- 実用性: そのまま使えるか、追加作業が必要か

### 15-2. レビュー・タイミング

- 即時レビュー: 外部影響あり（顧客対応、法務文書、資金移動）
- 日次レビュー: トレード日報、ブリーフィング内容
- 週次レビュー: Linear タスク進捗、Brain の draft → confirmed 昇格
- 月次レビュー: コスト実績、戦略と行動の整合性

### 15-3.「十分に良い」の定義

完璧を目指さない。以下を満たせば承認する:
- 目的を達成しているか → Yes
- 重大なエラーがないか → No
- 修正コストが新規作成より小さいか → Yes
→ 3つすべて該当すれば承認。該当しなければ差し戻し。

## 16. 並列思考 (Parallel Judgment)

### 16-1. 原則

Karpathy が指摘する「トークンスループットの最大化」。
エージェントを直列ではなく並列に動かし、結果を統合する。
1つのエージェントが30分かかるタスクを、
5つのエージェントに同時実行させれば6分で完了する。

### 16-2. Laa への適用

Phase 2 以降、以下を並列化する:
- CIO（市場データ収集）と CMO（コンテンツ下書き）を同時実行
- CTO（コード生成）と CQO（テストケース作成）を同時実行
- 複数の DPGE ループを異なるプロジェクトで同時進行

Phase 1 では、GenSpark での壁打ちと Claude Desktop での作業を
異なるテーマで並行して進めることが、この思考の練習になる。

## 17. 自己OSのアップデート戦略

### 17-1. 原則

エージェント組織のOSは Brain / HQ / Dev の3層。
このOS自体を定期的にアップデートしなければ、
市場の変化やツールの進化に取り残される。

### 17-2. アップデート・サイクル

- 週次: Brain の draft → confirmed 昇格、不要ファイルのアーカイブ
- 月次: CONTEXT.md の更新、ツールスタックの見直し、コスト実績の確認
- 四半期: workflow-philosophy.md の見直し、フェーズ進捗の評価、戦略の再定義
- 随時: 重大な気づき・新ツール・新概念の発見時は即座に Brain に記録

### 17-3. アップデートの判断基準

以下のいずれかに該当したら、該当ドキュメントを更新する:
- 現実の行動とドキュメントの記述が乖離している
- 新しいツールやフレームワークを採用した
- 失敗から新しい教訓を得た
- フェーズが移行した

## 18. 重要原則のまとめ

1. 第一原理で考える: 形ではなく機能から逆算する
2. 戦略は行動で測る: 時間の使い方が戦略そのもの
3. Delegate-Review-Own: 委任・検証・所有の境界を明確にする
4. Human-Above-the-Loop: 5%の判断に集中し、95%を委任する
5. Be Direct: AIへの指示は曖昧さゼロで、目標持続型にする
6. 書かれていない知識は存在しない: 24時間以内に Brain に転記する
7. 十分に良いで進む: 完璧を待たず、基準を満たしたら承認する
8. 並列で動かす: 直列思考を捨て、同時実行で速度を上げる
9. OS を更新し続ける: ドキュメントが古くなった瞬間、判断の質が劣化する
10. 失敗パターンを記録する: 同じ失敗を繰り返さないための最良の投資
11. エゴ/能力比率を1未満に保つ: エゴが能力を超えると、 現実からのフィードバック（トレードの損失、AIの出力、他者の意見）を 受け入れられなくなり、学習ループが壊れる。 タスクが崇高であれ地味であれ、必要なことをやる。
12. 最大限に真実を追求する: 自分に都合の良い解釈を避け、 シンプルで率直な言葉で現実を記述する。 これはAIへの指示品質にも、トレード日報の正直さにも直結する。
13. 観客ではなく参加者でいる: 情報を消費するだけの時間は観客席にいるのと同じ。 Brain に書く、コードを動かす、エントリーを判断する。 手を動かしている時間だけが「参加」。
14. 検証済みの知識だけを記憶する: Brain に書くのは成功と教訓のみ。 未検証の試行錯誤を蓄積すると、エージェントの判断基盤が汚染される。
